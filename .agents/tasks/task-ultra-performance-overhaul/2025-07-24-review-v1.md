# Ultra performance overhaul with projects enrichment, FAB + chatbot, and contact redesign

Four coordinated changes strip the Three.js / R3F / GSAP dependency tree (~945 KB), replace the WebGL background with CSS-only animated gradients gated on device capability, enrich the Projects section with six detailed projects, category filters, and a full-page detail overlay, introduce a Floating Action Button with an OpenAI-powered chatbot, and rebuild the Contact section with availability indicators, quick-contact cards, glassmorphism form, and a testimonial.

**Watch for:** The AI chatbot sends the user's OpenAI API key to `api.openai.com` from the browser — localStorage-stored secrets are trivially exfiltrable via XSS or browser extensions (confirmed). Chat history accumulates in localStorage with no size cap (confirmed). The `content-visibility: auto` applied to all `<section>` elements can hide in-viewport content during initial paint if `contain-intrinsic-size` underestimates actual section height (likely). External images (`pravatar.cc`, `picsum.photos`) are loaded at runtime with no fallback — if either CDN is down the hero/about/project screenshots are broken blank rects (confirmed).

## High-level view

The performance pivot removes three heavy dependencies (three, @react-three/fiber, @react-three/drei) and the manual chunk-splitting they required in vite.config, replacing the canvas-driven background with a CSS `@keyframes` animated radial-gradient that degrades to static or nothing depending on device capability. Lenis smooth-scroll is now skipped entirely on low-end devices. All runtime dependencies are now React 19, Framer Motion, Tailwind v4, Lenis, and Lucide icons.

The Projects section extracts its data into a shared module with six richly-typed placeholder projects, adds category filter pills, and delegates detail display to a `ProjectDetail` overlay rendered from App.tsx. The overlay locks body scroll, has an Escape key handler, and provides prev/next navigation. Image screenshots come from `picsum.photos` seed URLs — CDN-dependent with no fallback.

The chatbot lazy-loads from the FAB, persists its conversation in localStorage, and talks directly to `api.openai.com` with a user-supplied key stored in localStorage. There is no backend proxy, no rate limiting, and no key rotation — the threat model is limited to the key owner's own usage quota, but any XSS on the domain exfiltrates the key.

The Contact section gains quick-contact cards (WhatsApp with a hardcoded placeholder number, email, schedule-call), floating-label form with blur validation and character counter, error/retry state, and a testimonial. The WhatsApp card uses `bg-green-50` which breaks the dark theme pattern established elsewhere.

<summary>Issues (8)</summary>

1. **OpenAI key exposed client-side** — The API key is stored in localStorage and sent directly from the browser. Any XSS vector or malicious extension on the domain can exfiltrate it. If this ships publicly, document that users should use a scoped, low-limit key. (confirmed)
2. **Unbounded localStorage chat history** — `saveChatHistory` writes the full message array with no truncation. Long conversations will hit the ~5 MB localStorage limit, at which point writes silently fail. Add a rolling window or purge strategy. (confirmed)
3. **External image CDN dependency with no fallback** — Profile photos from `i.pravatar.cc` and project screenshots from `picsum.photos` have no `onerror` handler, placeholder, or local fallback. If either CDN is down or the seed URL contract changes, the UI shows broken images. (confirmed)
4. **`content-visibility: auto` on all sections risks layout shift** — `contain-intrinsic-size: auto 500px` underestimates sections like Contact (~1400px rendered height). Below-fold sections can flash or pop during scroll as the browser realizes actual height differs from the estimate. (likely)
5. **WhatsApp link uses placeholder phone number** — `https://wa.me/6281234567890` is clearly a placeholder; shipping this live would link visitors to a stranger's number. (confirmed)
6. **Hardcoded `bg-green-50` in quick-contact card** — The WhatsApp card icon wrapper uses `bg-green-50` / `group-hover:bg-green-100`, which won't adapt to dark theme CSS variable overrides. (confirmed)
7. **Chat message list keyed by array index** — Messages use `key={i}` which is stable here (append-only), but if a "delete message" feature is added later, React will reconcile incorrectly. Low risk for now. (possible)
8. **No abort controller for in-flight chat requests** — If the user closes the chatbot while a fetch is in progress, the response callback fires on an unmounted component. React 19 won't warn, but the state update is wasted and could surface errors. (likely)

## Client-side OpenAI integration and key exposure

The chatbot (`AIChatbot.tsx`) stores the user's OpenAI API key in localStorage under `porto-chat-api-key` and sends it as a `Bearer` token directly from the browser to `api.openai.com`. Any JavaScript executing on the same origin (XSS, injected analytics, browser extensions with host permissions) can read the key. Since this is a portfolio site with no user accounts, the blast radius is limited to the site owner — but if deployed on a shared subdomain (Vercel preview, GitHub Pages org), adjacent deployments might share storage origin.

`sendMessage` does not attach an `AbortController` to the fetch. If the user closes the chat panel mid-request, the response callback fires on a now-unmounted `AIChatbot` calling `setMessages`.

The chat history grows without bound:

```ts
// chatStorage.ts
export function saveChatHistory(messages: ChatMessage[]): void {
  try {
    localStorage.setItem(CHAT_HISTORY_STORAGE, JSON.stringify(messages));
  } catch {
    // Storage unavailable — silently drops
  }
}
```

Once the serialized history exceeds localStorage's ~5 MB budget, all subsequent writes silently fail (the catch swallows the `QuotaExceededError`). From the user's perspective, new messages appear in the UI during the session but vanish on reload.

## External image CDN dependency

Both Hero and About render profile photos from `https://i.pravatar.cc/300?img=68`. Project screenshots come from `https://picsum.photos/seed/{name}/800/500`:

```tsx
// Hero.tsx
<img src="https://i.pravatar.cc/300?img=68" alt="..." loading="lazy" ... />

// ProjectDetail.tsx — carousel images
{project.screenshots.map((src, i) => (
  <img src={src} alt={...} loading="lazy" ... />
))}
```

No `onError` fallback, no local placeholder, no skeleton shown while loading. If pravatar.cc returns a 503 (which it does periodically under load), the hero section shows a broken image icon where the profile photo should be. For a portfolio site, this is the most visible single point of failure.

## `content-visibility: auto` rendering budget

```css
section {
  content-visibility: auto;
  contain-intrinsic-size: auto 500px;
}
```

The Contact section renders well over 1000px. When the user first scrolls it into view, the browser commits the true height, causing a layout jump. The `auto` keyword helps after first paint (the browser remembers the real size), but the first visit still gets the shift. Consider per-section `contain-intrinsic-size` values or applying the rule only to sections with predictable heights.

## FAB performance mode lacks visual feedback

The performance mode cycles through `['full', 'battery-saver', 'low-gpu', 'reduced']` on each click with no visual indicator of the current mode beyond a title tooltip. A user who clicks it once has no immediate feedback about what changed. The theme button shows a Sun/Moon icon for its state; the performance button always shows a static Zap icon regardless of which mode is active.

## Dark theme quick-contact card gap

The WhatsApp quick-contact card uses:

```tsx
<div className="p-2.5 rounded-lg bg-green-50 group-hover:bg-green-100 transition-colors duration-300">
```

`bg-green-50` and `bg-green-100` are Tailwind static palette colors that won't respond to the `[data-theme='dark']` CSS variable overrides. In dark mode this renders a bright mint-green box on an otherwise dark surface. The Email and Schedule cards use `bg-terracotta/5` and `bg-sage/5` respectively, which properly inherit from CSS variables.
